## Learned

- **[high, self_checked]** TS `enum` is not erasable syntax (it compiles to a runtime object) and
  is rejected by Node's strip-only type stripping (`ERR_UNSUPPORTED_TYPESCRIPT_SYNTAX`), discovered
  by running the test suite, not by research. Worked around with a `const` object + derived union
  type (`src/env/types.ts`) — this constrains the whole project's TS style going forward: any future
  code (freeze mechanism, metric plumbing, eventually the world-model cell) must stick to
  Node-erasable TypeScript syntax unless/until a `typescript`-package build step is added.
- **[high, self_checked]** Confirmed directly (not from `notes/adr-0002-js-ml-stack.md`, which
  didn't test this): this Node version's type-stripping is enabled **by default**, not behind
  `--experimental-strip-types` — `process.allowedNodeEnvironmentFlags.has('--experimental-strip-types')`
  returns `true` but the *inverse* flag `--no-experimental-strip-types` is what's listed in
  `node --help`, meaning stripping is on unless explicitly disabled. Running a bare `.ts` file and
  `node --test` on `.ts` test files both worked with no flags. Recorded here since ADR-0002 hasn't
  formally named a test-tooling choice yet and this is now a concrete, verified data point for it.
- **[medium, self_checked]** Manhattan distance for both the view-radius check and the reward
  formula (rather than Euclidean or Chebyshev) makes exact-value test assertions straightforward,
  since every action changes it by exactly 1 — this made the reward and visibility tests exact-match
  rather than approximate/statistical, a stronger test than I'd first expected to be able to write
  for a stochastic-spawn environment.
