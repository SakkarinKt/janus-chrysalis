A second, smaller methodological finding while building the gradient-check test: naive
finite-differencing the straight-through estimator's *own* forward pass is identically zero,
because that forward value (the hard sample) is piecewise-constant in the logits by construction —
it's a separate discrete argument, not a continuous function of them. The gradient-check therefore
has to target `softmax(logits)` directly (what the estimator's backward pass is defined to
reproduce), not the estimator's own output. Added a companion test asserting the naive check *is*
zero, so this is a documented, tested fact rather than an unexplained oddity for the next person
who tries the obvious thing.

## Learned

- **[high, self_checked]** `tf.stopGradient` is not a function on `@tensorflow/tfjs-node@4.22.0`,
  `@tensorflow/tfjs-core@4.22.0`, or `@tensorflow/tfjs@4.22.0` — confirmed by direct execution
  (`typeof tf.stopGradient === "undefined"`; calling it throws). Corrects §4 of both
  `notes/adr-0002-js-ml-stack.md` and `notes/rssm-vs-ssm-implementation-robustness.md`, and the
  2026-07-17 ADR-notes entry that had (incorrectly) flipped §6's wording *to* `tf.stopGradient`.
  `tf.customGrad()` is the confirmed-working replacement and is what `straightThroughEstimator()`
  actually uses.
- **[high, self_checked]** A straight-through estimator's forward pass can't be finite-difference
  gradient-checked against itself when the forward value is a discrete sample supplied as a
  separate argument (as opposed to computed from the continuous input inside the same function) —
  the check is measuring a piecewise-constant function and will report zero everywhere regardless
  of correctness. The check has to target the soft/continuous path the backward gradient is
  defined to reproduce. Worth keeping in mind for any other STE-style code this project writes
  later (e.g. if a second discrete-latent mechanism is added).
- **[medium, self_checked]** This is the first ADR-0002-thread claim caught by actually running the
  pinned dependency's code rather than searching about it. Both times `tf.stopGradient` was
  "confirmed" (§4 of the ADR notes, and again in the 2026-07-17 correction), the confirmation was
  search-based — the guide page itself was 403'd on `WebFetch` per §4's own admission. No process
  change proposed this run (that would be a "Decisions needed"-adjacent judgment call about how
  much runtime verification every claim needs, which isn't this run's call to make unilaterally),
  but flagging the pattern in case it's useful signal for how much weight to put on
  `WebSearch`-only "confirmed present" claims about this specific dependency going forward.
