## Learned

- **[high, self_checked, primary-source]** Darwin support in `@tensorflow/tfjs-node`'s prebuilt
  native-addon pipeline was dropped project-wide after `3.6.1` (2021) and never restored through
  the current `4.23.0-rc.0` (still the `latest` dist-tag) — verified via the GCS bucket's own
  public object-list API, not inference from a single 404. This reframes PR #17/#18's finding from
  "the pinned version has a bug" to "the whole native-binding install path has been Linux/Windows-only
  for ~5 years," a materially larger and more durable risk than either prior stand-up recorded.
- **[high, self_checked]** node-pre-gyp's napi-tier resolution (highest declared `napi_versions`
  entry ≤ runtime napi) means this repo's Node pin (`engines.node >=22.18.0 <23`, which reports
  napi 10) structurally can't reach the older `napi-v7` tier where darwin builds do exist, even by
  pinning an old `tfjs-node` version — closing off "just use an older release" as a workaround too,
  not only "use the newest release."

  `node-pre-gyp install --fallback-to-build`'s from-source compile — which has real inputs to work
  with, since the arm64 `libtensorflow` C library *does* download fine via a hardcoded special-case
  URL in `install.js` — fails on the human's hardware (only verifiable there), or (b) a
