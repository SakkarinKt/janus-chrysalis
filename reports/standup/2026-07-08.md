- **[high, self_checked, primary-source]** `tensorflow/tfjs`'s last stable release is v4.22.0
  (2024-10-21) — confirmed via `registry.npmjs.org`'s raw JSON, no stable release in ~20 months.
  Commit activity continues but has slowed sharply (isolated commits with multi-month gaps), against
  246 open issues / 340 open PRs on the repo's own front page. Not dead, but low-cadence
  maintenance — a real risk factor for a multi-year research dependency.
- **[medium, self_checked, primary-source]** `tensorflow/tfjs` issue #8590 (open since 2025-09-13,
  unresolved): WebGPU backend produces all-zero gradients for a `simpleRNN` model on specific
  hardware (Pixel 10 Pro / Img-Tec GPU) — forward pass correct, backward pass silently broken. Not a
  blanket WebGPU-can't-train-RNNs claim (other WebGPU devices don't reproduce it), but concrete
  evidence the recurrent-gradient path on WebGPU is immature — directly relevant since an RSSM is
  fundamentally recurrent. Recommends CPU (`tfjs-node`)/WebGL over WebGPU for the Arm-A training
  path specifically, revisit WebGPU once this class of issue is resolved.
- **[high, self_checked, primary-source]** **Tooling correction, relevant beyond this note:**
  `WebFetch` is not blanket-403'd this session — it succeeded on `github.com` (repo/commits/
  releases/issues pages) and `registry.npmjs.org` (JSON API), and only 403'd on `tensorflow.org` and
  `npmjs.com`'s web page. The standing assumption in `notes/lit-map.md`'s tooling caveat (from PR
  #2's human reply) treats the 403 as a blanket persistent constraint; this session's evidence says
  it's domain-specific, not global. Recorded in the new note (§5) rather than editing `lit-map.md`
  this run (out of scope for one bounded increment; arXiv itself wasn't re-tested this session, so
  the original 403 finding there stands unchanged) — flagged here so the next run researching a

- **[medium, self_checked]** `tf.customGrad()` and `tf.stopGradient()` are stable, documented TF.js
  primitives sufficient to express an RSSM's straight-through/KL-balancing needs — feasibility risk
  for the RSSM op set is concentrated in backend choice (§2 above), not in whether TF.js's autograd
  system can express the model at all. No existing TF.js RSSM/Dreamer port was found in search —
  this would be a from-primitives implementation, which raises the Arm-A milestone's effort estimate
