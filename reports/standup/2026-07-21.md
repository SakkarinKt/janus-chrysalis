## Learned

- **[high, self_checked, execution-confirmed]** Differentiating `RSSMCell.step()`+`.prior()`
  chained across ≥2 timesteps crashes `tf.variableGrads` inside `tfjs-layers`' RNN backward pass.
  A single timestep works. Root cause: an RNN layer's hidden output feeding back into its own next
  call's feature input (not just `initialState`) within one differentiation trace — exactly
  `RSSMCell.step()`'s design, since `z_{t-1}` (needed for step `t`'s input) comes from `.prior()`
  applied to step `t-1`'s hidden state.
- **[high, self_checked, primary-source]** This matches `tensorflow/tfjs#1529` (2019) and `#3550`
  (2020) almost exactly — both report the identical error for single-timestep RNN training, with
  a reported workaround (feed ≥2 timesteps per `.apply()` call) that doesn't transfer here, since
  RSSM's recurrence can't assemble its input sequence before running the RNN layer (each step's
  input depends on the previous step's *output* via the stochastic head). Confirmed still
  reproducing on this repo's pinned `tfjs-node@4.22.0`, seven years after #1529 was filed —
  read as "unlikely to be fixed upstream on any timeline this project can wait for," not as "will
  probably land soon."
- **[medium, self_checked]** `tf.variableGrads(f)` with no explicit `varList` defaults to *all*
  currently-registered trainable variables process-wide, which — in a test file where many
  `RSSMCell`s accumulate variables across tests (tfjs never garbage-collects them) — throws inside
  tfjs's own gradient bookkeeping. Passing an explicit `varList` (diffed from `tf.engine()
  .registeredVariables` before/after the cell's first `.apply()` call) avoids this; useful pattern
  for any future gradient-check test in this file.
- **[medium, self_checked]** Re-confirmed the STE forward-pass-is-piecewise-constant trap (already
  documented for the standalone estimator, PR #20) generalizes: an initial attempt at the
  single-step end-to-end check built its loss from `.prior()`'s `sample` output and got a spurious
  0-vs-nonzero mismatch for the same reason. Fixed by building the loss from `probs` (plain
  softmax, no STE) instead — the STE's own gradient correctness is already covered by the existing
  standalone tests, so this new test's job is checking the GRU/dense path behind it, not
  re-litigating the STE.

## Blockers

- Multi-step differentiable training (BPTT) through `RSSMCell` is blocked by the tfjs-layers bug
  above. This affects any world-model loss that needs gradients through more than one recurrent
  step — which real training will, per proposal `0001`'s design (KL balancing, observation
  reconstruction over a sequence chunk, item 3 in `loop/GOAL.md`'s priority list). Not something
  this loop can fix unilaterally (see "Decisions needed").
