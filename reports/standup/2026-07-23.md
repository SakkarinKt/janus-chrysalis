## Learned

- **[medium, self_checked, execution-confirmed]** Multi-step BPTT throughput on this stack, measured
  for the first time: `modelTimestepsPerSec` (chain length × batch × gradient-steps/sec) stays
  roughly flat (~1,700–1,900/sec, freshest run) across every chain length swept (2 through 32) —
  expected, since RSSM forward/backward compute is ~linear in chain length and per-call
  wrapper/dispatch overhead is a small, roughly constant fraction of it. On raw stack throughput,
  ADR-0002 decision 5's kill criterion does not fire. Medium, not high, confidence: only 30 timed
  iterations per chain length (an order-of-magnitude read), and this measures RSSM forward/backward
  cost in isolation — no replay buffer, no real losses, no actual environment stepping yet.
- **[medium, self_checked]** Converting stack throughput to *environment*-steps/sec — the unit
  proposal `0001`'s overnight-run budget is denominated in — needs a replay ratio (how many times
  each collected transition is replayed through a gradient step, on average) that isn't fixed
  anywhere in this repo yet. At replay ratio 16: ≈30 min/arm/seed for 200K env steps (comfortably
  overnight). At replay ratio 512: ≈16 h (right at the edge of "overnight"). Both are plausible
  training configurations, so this isn't resolved by today's measurement alone — it's a decision to
  make before the real Arm-A training runs, not a blocker for now.
- **[medium, self_checked]** The single-step gradient figure has moved on every run of this script so
  far (≈101/s on 2026-07-21; ≈57/s and then ≈90/s across two runs today), with no changes to the
  forward/backward arithmetic on that path between those dates. Container-to-container CPU
  variance, not a code regression — flagged explicitly in both docs updates so the absolute numbers
  aren't read as a portable hardware spec.
- **[high, self_checked]** Both halves of ADR-0002 decision 5's hard kill criterion — gradient
  correctness (2026-07-22) and raw steps/sec (today) — evaluated and neither fired. The week-3

## Correction (2026-07-24, appended — original report above left unchanged)

Two claims above (`Done`'s replay-ratio paragraph, `Learned`'s third and fourth bullets) don't hold
up, per @SakkarinKt's PR #24 review the same day this report was written:

- **"Silently assumed replay ratio 1" was backwards.** Dropping the batch dimension (`BATCH = 16`)
  from the original conversion is arithmetically `modelTimestepsPerSec ÷ BATCH`, which is what
  environment-steps/sec equals at replay ratio = `BATCH` (16) — not ratio 1. The pre-correction
  ~70–83 figure was a conservative (~16×-too-low) read of raw stack throughput, coincidentally valid
  at replay ratio 16 specifically, not an optimistic no-replay assumption. The replay-ratio bracket
  numbers themselves (≈30 min at ratio 16, ≈16 h at ratio 512) were independently verified and stand.
- **"No change to the forward/gradient arithmetic between those dates" (2026-07-21 vs. today) is
  wrong.** `RSSMCell.step()` changed from a `tf.layers.rnn`-wrapped `.apply()` call to a direct
  `cell.call()` between the 2026-07-21 run and today's two runs (§9 of
  `notes/adr-0002-js-ml-stack.md`, landed 2026-07-22). The valid same-code comparison is between
  today's two runs only (~57/s vs. ~90/s), which is consistent with container variance.
- A `tf.variableGrads` tensor leak (the `grads` return value was never disposed, only `value`) in
  this report's benchmark code was also caught in the same review and fixed today.
