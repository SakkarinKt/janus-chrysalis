## Learned

- **[high, self_checked]** `@tensorflow/tfjs-node@4.22.0`'s native binding is fine on Node ≥23 —
  the oneDNN banner shows in the failing log even there — but its JS wrapper calls the
  runtime-deprecated-then-removed `util.isNullOrUndefined` (DEP0051), so every tensor op throws
  `TypeError: is not a function` on Node 23+. The break is pure JS, not the native build; confirmed
  by reproducing the reviewer's diagnosis locally on Node 22.22.2 (in-range) rather than just
  trusting the PR comment.
- **[high, self_checked]** Local `npm test` green is not the same claim as CI green — this run's
  own predecessor (PR #15) shipped with a passing local run while CI was actually red for two
  reasons (`npm ci` missing from the `test` job; the Node-version-range/DEP0051 interaction), both
  latent since earlier PRs and only surfaced by the first dependency-importing test. Taking this
  as binding process feedback: future runs check the PR's own CI status, not just local results,
  before calling a run done.
- **[medium, self_checked]** When a pre-approved boundary (the tfjs dependency carve-out's "own
  PR touching only `package.json` + lockfile + smoke import") turns out to make CI unfixable
  in-scope — here, `npm ci` needs the lockfile the same PR introduces, so the fix couldn't
  plausibly have landed as an earlier, separate change — the right move is to flag the conflict
